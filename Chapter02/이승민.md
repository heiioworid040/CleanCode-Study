
# 📖 책 내용 정리 

## 🔶 2장 의미 있는 이름  

### 1️⃣ 의도를 분명히 밝혀라 
- 변수 (혹은 함수나 클래스)의 존재 이유, 수행기능, 사용방법이 모두 나타나야한다.
  ( 코드의 맥락이 코드 자체에 명시적으로 드러나야함 )
- 따로 주석이 필요하다면 의도를 분명히 드러내지 못했다는 것 

##### ◾ 안 좋은 예 

<br>

``` 
int tmp 
string a
int answer 
int getSum 
``` 

``` 
public List<String> getNum(int[] num){
    List<String> tmp = new ArrayList<String>();
    for(int i=0; i<num.length(); i++){
        if(num.length()!=0) tmp.add(num[i]);
    }
    
    return tmp ;
}
``` 



<br>

##### ◾ 좋은 예

<br>


``` 
int salesRate
string fileName
int getDepartmentScore
``` 

``` 
public List<String> getBoardNumber(int[] num){
    List<String> BoardNumber = new ArrayList<String>();
    for(int i=0; i<num.length(); i++){
        if(num.length()!=0) BoardNumber.add(num[i]);
    }
    return BoardNumber;
}
``` 

- 함수 사용 시에도 해당 함수가 어떤 일을 하는지 이해하기 쉽도록 좀 더 명시적인 함수를 사용하는 것이 좋다. 



<br>

### 2️⃣ 그릇된 정보를 피하라 
- 약어 사용, 최종 return값 인 것 같은(혼선을 주는) 변수명, 서로 흡사한 이름, 일관성이 떨어지는 이름 자제 
- 소문자 L, 대문자 O 같이 혼선을 주는 변수 명 자제  

<br>

### 3️⃣ 의미 있게 구분하라  
- 변수에 연속 된 숫자 덧붙이지 않기  ( `num1, num2 ... ` )
- 접두어 자제 (a, the, an ... )
- 이름에 Table, String 등을 넣지 않기

```
getAccount();
getAccounts();
getAccountInfo();
int num1;
int num2;
string theMessage;
int numberInt;
```

<br>

### 4️⃣ 발음하기 쉬운 이름을 사용하라 
- 발음하기 어려운 이름은 토론하거나 새로운 사람이 들어오면 설명 해 주기도 어렵다.

<br>

### 5️⃣ 검색하기 쉬운 이름을 사용하라 
- 문자 하나를 사용하는 이름과 상수는 텍스트코드에서 쉽게 눈에 띄지 않는다.
- 영어에서 가장 많이 쓰는 문자 e를 넣는것도 변수이름으로 적합하지 못하다. 
- 간단한 메서드에서 로컬 변수만한 문자를 사용한다. (이름 길이는 범위 크기에 비례 해야 한다.)

<br>

### 6️⃣ 인코딩을 피하라 
- 유형이나 범위 정보까지 인코딩에 넣으면 이름을 해독하기 어려워진다.

##### ◾ 헝가리식 표기법
- 과거 윈도 C API는 컴파일러가 타입을 점검하지 않았으므로 프로그래머에게 타입을 기억할 단서가 필요했으나,
  최근에는 컴파일러가 타입을 기억하고 강제하고, 클래스와 함수는 점차 작아지는 추세이기 때문에 헝가리식 표기법이나
  기타 인코딩 방식이 오히려 방해가 된다.

<br>

##### ◾ 멤버 변수 접두어
- 멤버변수에 `m_` 과 같이 접두어를 붙이는 것. 하지만 이제는 접두어를 붙일 필요가 없다. 


<br>

##### ◾ 인터페이스 클래스와 구현 클래스 
- 때로는 인코딩이 필요한 경우도 있다. 
- 인터페이스 이름은 접두어를 붙이지 않는 것이 좋다. 
- 인터페이스 클래스 이름과 구현 클래스 이름 중 하나를 인코딩 해야 한다면 구현 클래스 이름을 택하는 것이 좋다. 

<br>

### 7️⃣  자신의 기억력을 자랑하지 마라 
- 전통적으로 반복 횟수 변수는 한 글자를 사용하기 때문에 반복 횟수를 세는 `i,j,k`는 상관없지만 그 외에는 한 글자로 변수를 생성하는 것은 적절하지 않다.
- 프로그래머는 명료함이 최고이다. 남들이 이해하는 코드를 만들어야한다. 

<br>

### 8️⃣ 클래스 이름
- 클래스 이름과 객체 이름은 명사나 명사구가 좋다.
ex) Cousomer, Account, AddressPaper 
→ Manager, Processor, Data, Info는 피하기 

<br>

### 9️⃣  메서드이름
- 동사나 동사구가 적합하다.
ex) postPayment, deletePage, save 등
- 접근자, 변경자, 조건자는 각 get, set, is를 붙여주면 좋다. 
- 생성자를 중복 정의 할 때에는 정적 팩토리 메서드를 사용한다.
  메서드는 인수를 설명하는 이름으로 사용한다. 
- 생성자를 제한하려면 해당 생성자를 private로 선언한다. 

<br>

### 1️⃣0️⃣ 기발한 이름은 피해라 
- 구어체나 속어 사용 금지. 의로를 분명하고 솔직하게 표현하는 것이 좋다. 

<br>

### 1️⃣1️⃣ 한 개념에 한 단어를 사용하라
- 이클립스, 인텔리제이 등과 같은 최신 IDE는 문맥에 맞는 단서를 제공한다.
- 메서드 이름은 독자적이고 일관적이어야한다.
ex ) Controller, Manager를 섞어쓰면 혼선이 생기므로 일관성있는 어휘를 사용한다.


<br>


### 1️⃣2️⃣  말장난을 하지마라 
- 한 단어를 두가지 목적으로 사용하지 마라
ex ) add라는 메서드가 기존 값 두개를 더해서 새로운 값을 만드는 메서드로 사용하였으나, 같은 맥락이 아닌데도 일관성을 고려해 add라는 이름을 선택하면 안된다. 맥락이 다르다면 insert 등의 더 적절한 이름을 택해야한다.

<br>


### 1️⃣3️⃣ 해법 영역에서 가져온 이름을 사용하라
- 코드를 읽을 사람도 프로그래머다. 프로그래머에게 익숙한 기술 개념을 사용한다.
ex) Queue

<br>


### 1️⃣4️⃣ 문제 영역에서 가져온 이름을 사용하라 
- 적절한 프로그래머 용어가 없다면 문제 영역에서 이름을 가져온다. 

<br>

### 1️⃣5️⃣ 의미 있는 맥락을 추가하라  
- 함수 이름은 맥락 일부만 제공하고 알고리즘이 나머지 맥락을 제공한다.
- 맥락을 개선하면 함수를 쪼개기가 쉬워지고 알고리즘도 더 명확해진다. 

<br>

### 1️⃣6️⃣  불필요한 맥락을 없애라
- 의미가 분명한 경우에 한애서 짧은 이름이 긴 이름보다 좋다.

<br>

### 1️⃣7️⃣ 마치면서 
- 사람들이 이름을 바꾸지 않으려는 이유는 다른 개발자가 반대할까봐 두려워서이다. 하지만 좋은 이름으로 바꿔주면 반갑고 고맙다.
- 표나 자료 구조처럼 읽히는 코드를 짜는 데 집중해야한다. 
- 다른사람이 짠 코드를 손본다면 리팩터링 도구를 사용해 문제 해결 목적으로 이름을 개선하면 단기적인 효과는 물론 장기적인 이익도 보장된다. 

<br>

# 💬 소감
항상 하는 생각은 **코드짜는 것 보다 이름 짓는게 더 힘들다**였다.
그래서 이 챕터를 읽으면서 많은 공감이 되면서도 반성도 많이 했다.
한번씩 난감한 이름을 지어야하는 경우가 많은데 (예를 들어 한국어자체가 아주 긴... 해외 협력사 사외 적치장 사용률 (예시일뿐이다.) 이런 이름을 클래스나 메서드로 지을때
이름만 30분 넘게 고민한 기억이 있다.
길게 지으니 이게 맞나싶고, 짧게 짓자니 이 이름을 보고 뭔지 알 수가 없었기 때문이다.
그래도 불분명한것보단 긴게낫지 하는 생각으로 길게 지었지만,, 이걸 다른 개발자가 봤을 때 이해 할 수 있느냐까지는 생각을 못했던 건 사실이다.
코드를 짜는 것도 개발자고, 그 코드를 읽는 것도 개발자이다.
너무 단어를 단편적으로 보는 것이 아니라 개발자들 사이에서 사용하는 개념까지 사용하여 쉽게 표현 할 수 있다는 것을 깨달았다.
다음 메서드나 클래스 이름을 지을 때, 오늘 공부 했던 내용들을 잘 참고해서 더 명확한 이름을 지을 수 있기를 기대 해 본다.
